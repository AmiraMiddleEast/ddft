    2. Create `proxy.ts` at repo root (NOT `middleware.ts` — per P-01, Next 16 renamed this file) with this EXACT content (per P-08 cookie-presence pattern — avoids the Edge vs Node runtime question entirely):

       ```typescript
       import { type NextRequest, NextResponse } from "next/server";

       // better-auth default session cookie name.
       // ASSUMED A2: if advanced.cookiePrefix is set in lib/auth.ts, update here.
       // Current lib/auth.ts does not set cookiePrefix, so default applies.
       const SESSION_COOKIE = "better-auth.session_token";

       export function proxy(req: NextRequest) {
         const hasSession = req.cookies.has(SESSION_COOKIE);
         const { pathname } = req.nextUrl;
         const isLoginPage = pathname.startsWith("/login");

         // Unauthenticated → /login (except on /login itself)
         if (!hasSession && !isLoginPage) {
           return NextResponse.redirect(new URL("/login", req.url));
         }

         // Authenticated → bounce away from /login back to /
         if (hasSession && isLoginPage) {
           return NextResponse.redirect(new URL("/", req.url));
         }

         return NextResponse.next();
       }

       export const config = {
         // Match all routes except: API routes, Next internals, static files, favicon
         matcher: ["/((?!api|_next/static|_next/image|favicon.ico).*)"],
       };
       ```

       Key design points (do NOT change without reading P-08):
       - Function is named `proxy` (NOT `middleware`) per Next 16 rename
       - File is `proxy.ts` (NOT `middleware.ts`)
       - No `runtime: "nodejs"` needed because we don't hit the DB from here — just check cookie presence
       - Authoritative session check happens in Plan 04's `app/(app)/layout.tsx` (per P-02 defense-in-depth)
       - Matcher excludes `api|_next/static|_next/image|favicon.ico` so auth endpoints and static assets are NOT redirected

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → /api/auth/* | Unauthenticated POSTs to sign-in (rate-limited); sign-out needs a valid session cookie |
| client → /(protected) | proxy.ts checks cookie presence; layout.tsx will verify session server-side (Plan 04) |
| app → DB | better-auth uses parameterized Drizzle queries — no raw SQL |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | `/api/auth/sign-in/email` — credential stuffing / brute force | mitigate | `rateLimit.customRules["/sign-in/email"]: { window: 60, max: 5 }` in lib/auth.ts (Task 1, D-15); `minPasswordLength: 12` (D-14) raises brute-force cost |
| T-03-02 | Spoofing | Public signup creating unauthorized accounts | mitigate | `emailAndPassword.disableSignUp: true` in lib/auth.ts (Task 1, D-10) blocks `/api/auth/sign-up/email` |
| T-03-03 | Info Disclosure | Password hashing weakness | mitigate | better-auth uses scrypt by default (constant-time compare, salted) — no hand-rolled hashing (per §Don't Hand-Roll) |
| T-03-04 | Info Disclosure | Session cookie theft via XSS | mitigate | better-auth sets session cookie `httpOnly, Secure, SameSite=Lax` by default; React auto-escapes output (no raw dangerouslySetInnerHTML in this plan's files) |
| T-03-05 | Tampering | CSRF on login/logout POST | mitigate | better-auth default SameSite=Lax + origin check on API routes (per §Don't Hand-Roll) |
| T-03-06 | Info Disclosure | BETTER_AUTH_SECRET leak | mitigate | Task 1 throws at startup if env var missing; `.env.local` is gitignored (Plan 01); secret never imported into client files (lib/auth.ts is server-only) |
| T-03-07 | Tampering | Proxy matcher bypass | mitigate | Proxy is UX-only, not authoritative (per P-02). Plan 04's `app/(app)/layout.tsx` calls `auth.api.getSession` as the real gate — defense in depth |
| T-03-08 | Elevation of Privilege | DB adapter schema drift → silent auth bypass | mitigate | Schema generated by better-auth CLI (Task 2), never hand-written (per P-06); drizzle-kit push verifies tables match |
| T-03-09 | Info Disclosure | SQL injection via sign-in params | mitigate | Drizzle ORM uses parameterized queries; no raw SQL in this phase (per §Security Domain V11) |
</threat_model>
